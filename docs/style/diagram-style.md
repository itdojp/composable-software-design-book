# Diagram Style Guide

## Scope

This guide standardizes the visual and editorial rules for commutative diagrams, string diagrams, and architecture diagrams.
Use it together with `docs/style/notation.md`.

## Shared Rules

- Every diagram must answer one review question or support one explicit claim.
- Every diagram must use canonical labels from the glossary and term base.
- Every diagram must be referenced in the surrounding prose before the reader encounters it.
- Every diagram caption must state why the diagram exists, not just what it contains.

## Commutative Diagrams

- Use commutative diagrams when multiple paths are claimed to preserve the same meaning.
- Label arrows with transformation names rather than implementation detail.
- State the preserved invariant in the prose directly before or after the figure.
- If the diagram is only illustrative and not a proof obligation, say so explicitly.

## String Diagrams

- Use string diagrams for sequential and parallel composition, orchestration, or workflow reasoning.
- Keep the direction of flow consistent within a chapter.
- Make concurrency and synchronization points visually explicit.
- Name recovery or fallback edges when failure handling matters.

## Architecture Diagrams

- Use architecture diagrams for bounded contexts, interfaces, external systems, and operational boundaries.
- Show the stable interfaces and contracts, not every runtime detail.
- Reuse the same artifact names that appear in the text and running example.
- Avoid diagram labels that duplicate whole sentences.

## Figure Captions

- Use the format `Figure N.M. Short declarative title.` for numbered figures.
- Follow the title sentence with one short `> **Reader takeaway.** ...` callout sentence before the figure body.
- Use a publication-grade image asset for reader-facing core figures when `project-management/figure-program.md` defines one.
- Use a Mermaid block only when a chapter needs a lightweight local sketch or no publication-grade redraw exists yet.
- If a diagram supports an invariant, name the invariant in the caption or adjacent prose.
- Claim IDs may appear only when they already exist in `verification/traceability-matrix.md`.
- Reader-visible claim IDs belong in figure captions and chapter-level review callouts, not in ordinary body paragraphs or diagram node labels.
- Publication-grade redraws are generated by `scripts/render-publication-figures.py` and inventoried in `project-management/figure-program.md`.

## Complex Image Accessibility

- Treat a figure as complex when it carries a relationship, comparison, sequence, proof obligation, or conclusion that the surrounding prose does not already state in full.
- Keep the caption concise: it identifies the figure and its claim, but it is not a substitute for an equivalent text description.
- Keep Markdown alt text short enough to identify the figure's subject and purpose in its immediate context; do not duplicate the caption or the full long description.
- Place a visible `**Long description — Figure N.M.**` block immediately after every reader-critical complex image.
- In the long description, name the important nodes or regions, explain their relationships in reading order, and state the comparison or conclusion that the image supports.
- Do not rely on color, position, line style, or arrow shape without naming the meaning in text.
- Give every standalone publication SVG `role="img"` and an `aria-labelledby` reference to one non-empty, unique `title` and `desc` pair.
- Use the SVG title as the accessible name and the SVG description as a concise structural summary; keep the adjacent long description as the detailed equivalent text for the chapter's argument.
- Generate SVG metadata from the same figure specification as the visible nodes and edges, and reject missing, empty, placeholder, duplicate, or dangling metadata during QA.
- Review the caption, alt text, adjacent long description, and standalone SVG metadata as four distinct deliverables before accepting a reader-critical figure.

## Invariant Mapping

- When a diagram encodes an invariant, restate that invariant in prose.
- Link the invariant to the related artifact, review checkpoint, or verification evidence.
- Keep the invariant wording stable across chapters unless the scope changes intentionally.
